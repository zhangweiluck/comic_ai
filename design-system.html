01 / Principles

设计原则

克制即品牌

界面不依赖渐变、照片、厚重阴影或营销式装饰。近黑画布、白色细线和精准排版构成主要识别。

形状系统单一

所有交互元素使用 pill。卡片只使用 8px 小圆角，避免让界面变成松散的圆角后台模板。

排版承担情绪

大字号、常规字重、mono eyebrow 和清晰留白负责建立研究实验室般的高级感。

Token Group Scale Usage Implementation Rule
Spacing 2, 4, 8, 12, 16, 24, 32, 48, 64px Page rhythm, card padding, control gaps Use 4px base; avoid one-off spacing.
Radius 0, 8px, 9999px Bands, cards, pills Cards stay 8px; interactive controls use pill shape.
Elevation Flat, hairline Depth by borders No shadows; use 1px borders and surface contrast.
Grid 1200px container; 2-up desktop; 1-up mobile Marketing and product surfaces Collapse before content becomes cramped.
Icon 16, 20, 24px Buttons, nav, metadata Line icons, single weight, no filled decorative icons.

Icon System

Use line icons with 1.75px stroke, round caps/joins, 16/20/24px sizes. Icon-only controls must use an accessible label.

Icon Rules

  • Preferred source: lucide-style outline icons or an equivalent single-stroke library.
  • Default size: 20px inside 40px icon buttons; 16px for dense metadata; 24px for empty states.
  • Do not mix filled, duotone, emoji, and outline icons in the same product surface.
  • Pair icons with text for unfamiliar actions. Icon-only is reserved for repeated, common controls.
  • Optical alignment: center icons inside the pill and avoid resizing the button on hover/loading.
Token Layer Naming Pattern Examples Use Rule
Primitive primitive.color.*, primitive.space.* primitive.color.neutral.950 = #0a0a0a, primitive.space.4 = 16px Raw values. Do not use directly in product components except inside theme mapping.
Semantic color.bg.*, color.text.*, color.border.* color.bg.canvas, color.bg.surface, color.text.primary, color.border.subtle Default layer for application UI. Semantic tokens describe intent rather than raw color names.
Component component.[name].[part].[state] component.button.outline.border, component.input.error.border, component.card.radius Use when a component needs stable theming independent of the global semantic layer.
State state.[intent].* state.error.text, state.warning.border, state.success.icon Use for validation, alerts, badges, and task status. Never rely on color alone.

导航组件 Sidebar, Topbar, Menu, Tabs, Breadcrumb, Pagination, Stepper Current, expanded, collapsed, selected, disabled, overflow, keyboard focus Use semantic landmarks. Menus and tabs need arrow-key behavior when implemented as composite widgets.
反馈组件 Toast, Alert, Modal, Drawer, Tooltip, Popover, Loading, Progress Open, closed, entering, exiting, loading, success, warning, error, blocking/non-blocking Dialogs trap focus. Toasts use live regions. Tooltips are advisory only and must not contain required actions.
数据展示 Table, Card, List, Collapse, Empty State, Statistic, Chart Container, Timeline Loading, empty, populated, sorted, filtered, selected, partial failure, error Tables must define narrow-screen behavior. Charts require text summaries and never communicate status by color alone.
页面模板 Login, Dashboard, Settings, Profile, List Page, Detail Page, Form Page, Workspace Loading, empty, blocked, editable, read-only, error, success Templates compose blocks; they should expose data slots and not hide domain logic inside visuals.
Contract Area Required Convention Example Review Check
Component naming PascalCase for components, kebab-case for CSS classes, semantic variant names. Button, ProjectListPage, .button-primary, variant="danger" No visual-only names such as BigBlueButton or Card2.
Slots and parts Expose stable slots for compound components. data-slot="trigger", data-slot="content", data-slot="footer" Consumers can customize structure without reaching into private markup.
State styling Use data attributes for visual states and ARIA for accessibility states. data-state="open", data-disabled, aria-expanded="true" State is testable without depending on generated class names.
Token usage No hardcoded colors, spacing, radius, or shadows inside components. color: var(--color-text-primary), gap: var(--space-md) Style diffs should reference tokens except one-off demos in documentation.
Accessibility Every interactive control has role/name/state; dialogs and popovers handle focus. aria-label, aria-describedby, role="dialog" Keyboard-only path covers open, operate, close, and recovery.
Testing Each component gets render, state, keyboard, disabled, and error-state tests. Button disabled prevents submit, Dialog returns focus Tests assert behavior and accessible names, not only snapshots.

Definition Of Done

  • Uses semantic tokens, not raw hex values, in production component CSS.
  • Documents when to use the component and when not to use it.
  • Includes default, hover, focus, disabled, loading, empty, and error states where relevant.
  • Passes keyboard and screen-reader checks for its interaction role.
  • Defines responsive behavior before implementation lands.

组件状态规范

Component Required States Do Avoid
Button Default, hover, active, focus, disabled, loading, primary, outline, danger, icon-only, full width Keep shape pill; show focus ring; disable duplicate submissions. Mixing rectangles and pills in the same action family.
Input Empty, filled, focus, error, disabled, readonly, helper text, validation message Use field-level messages and `aria-describedby`. Only showing errors through color.
Modal Closed, opening, open, submitting, error Trap focus, close on Escape, return focus to trigger. Leaving background controls keyboard-focusable.
Table Loading, empty, populated, sorted, filtered, selected, error Preserve header visibility and provide empty recovery actions. Rendering blank surfaces with no explanation.
Toast Info, success, warning, error, dismissing Use polite live regions for non-critical feedback. Using toast for blocking decisions that require confirmation.
State Button Input / Select Modal / Drawer Table / List

Hover Canvas-soft fill; no size shift. Border becomes stronger; no layout shift. Only actionable controls change; surface remains stable. Row background softens; selected state remains distinguishable.
Focus 2px focus ring using --focus, offset 3px. Focus ring plus aria-describedby when helper/error exists. Focus trapped inside; Escape closes non-destructive dialogs. Keyboard row focus visible; sortable headers announce sort state.

05 / Patterns

模式与使用规范

Form Validation

Inline errors for field problems, toast for global failures, disabled state for unmet preconditions.

Destructive Confirm

Danger actions require a confirmation dialog. Never hide destructive commands inside visual primary buttons.

Operational Gate

Permissions, membership, credits, and missing data should block action with a precise reason and next step.

Do

  • Use near-black canvas as the only page background.
  • Use mono uppercase labels for section eyebrows and technical metadata.
  • Let hairline borders carry structure.
  • Keep interaction shapes consistent.

Don't

  • Do not introduce light mode for this kit.
  • Do not add shadows, glossy cards, gradients, or photo-heavy heroes.
  • Do not use filled CTAs broadly.
  • Do not rely on color alone for validation or status.

06 / Responsive

响应式规则

Viewport Breakpoint Layout Navigation Data Display
Desktop ≥ 1100px Persistent side navigation, 1200px content container, 2-4 column grids. Sidebar remains visible; topbar keeps primary actions inline. Tables use full columns, sticky headers optional for dense pages.
Tablet 768px - 1099px Single app column or 2-column content where density remains readable. Sidebar moves above content or becomes compact rail. Tables may horizontally scroll; preserve first column and row actions.
Mobile < 768px One-column layout, reduced section padding, display type clamps to 48px or below. Navigation collapses to menu button or top segmented control; touch target min 44px. Dense tables become card lists when comparison is not essential; otherwise horizontal scroll with visible affordance.

Modal To Drawer

On desktop, dialogs may center with constrained width. On mobile, complex forms should become bottom or full-screen drawers with sticky footer actions.

Touch Targets

Interactive targets must be at least 44 × 44px on touch devices. Icon-only buttons require accessible labels and visible focus state.

Text Rules

Do not scale text with viewport width. Use explicit breakpoint sizes and line lengths; keep long words from overflowing controls.

09 / Component System

设计系统 → 组件系统开发指南

把设计师给出的视觉规范转化成稳定、可复用、可组合、可治理的前端组件体系。目标不是照着设计稿写 UI，而是建立一套能长期支撑产品演进的代码资产。

一致性

同一种视觉和交互在全局表现一致，避免页面各自发明按钮、表单和反馈。

复用性

高频 UI 由组件资产承载，业务开发组合组件而不是重复造控件。

可维护性

修改 token 或基础组件即可影响全局，而不是在页面里四处改样式。

可演进性

主题、国际化、响应式和业务扩展都基于稳定边界演进，不推倒重来。

Layer Responsibility Examples Must Not Do Outputs
Design Tokens 把颜色、字号、间距、圆角、边框、层级、动效、断点转成代码原子。 color.bg.canvas, space.md, radius.control, motion.fast 不要在组件里使用裸色值、魔法间距或一次性圆角。 tokens.ts, theme.css, Tailwind/Figma 映射, 禁用裸值规范。
Primitive Components 无业务语义的基础 UI 控件，承担状态、键盘、ARIA、表单兼容和响应式基础表现。 Button, Input, Select, Checkbox, Modal, Tooltip, Tabs, Badge, Avatar, Spinner 不要绑定业务提交、权限判断、订单状态或具体 API。 源码、Props 类型、Storybook、基础交互测试、可访问性检查。
Composite Components 由多个 Primitive 组成的通用业务无关模式，解决场景一致性。 PageHeader, EmptyState, DataTable, FilterPanel, ConfirmDialog, FormSection, UploadPanel 不要直接请求业务 API；不要把用户、订单、计费等业务语义写入 shared 组件。 复合组件库、插槽规范、典型页面模板、复用场景说明。
Feature Components 带明确业务语义的模块，封装业务 UI、状态、hooks、API 和错误处理。 UserPermissionEditor, OrderTable, BillingSummary, InvoicePreview 不要污染 Primitive/Composite；不要把页面写成业务逻辑堆叠。 Feature 目录、业务 hooks、业务类型、API 封装、页面编排代码。

1. 判断层级 Token / Primitive / Composite / Feature / Page internal。 组件归属说明。 业务语义不得进入 shared Primitive。
2. 定义职责 一句话说明负责什么、不负责什么。 组件 README 或 Storybook notes。 边界不清不得实现。

7. 写使用规范 什么时候用、什么时候不用、错误用法、可访问性说明。 组件文档。 必须写“不要怎么用”。

Primitive API Standard

type ButtonVariant = "primary" | "secondary" | "ghost" | "danger";
type ButtonSize = "sm" | "md" | "lg";

type ButtonProps = {
  variant?: ButtonVariant;
  size?: ButtonSize;
  loading?: boolean;
  disabled?: boolean;
  fullWidth?: boolean;
  children: React.ReactNode;
};

组件 API 表达用途语义，不表达视觉细节。Variant 来自设计系统，不自由发挥。

Composition Over Configuration

<Card>
  <Card.Header>
    <UserInfo />
  </Card.Header>
  <Card.Body>...</Card.Body>
  <Card.Footer>
    <Button>Edit</Button>
  </Card.Footer>
</Card>

长期来看，组合能力比把所有场景塞进 props 更健康。

Governance Rule 1

按钮必须用 Button，输入框必须用 Input，弹窗必须用 Modal，颜色和间距必须用 Token。禁止绕过组件库。

Governance Rule 2

新增组件前先判断已有组件能否满足；是新 variant、Composite，还是 Feature 内部组件。

Governance Rule 3

定期复盘高频组件、无人使用组件、props 过多组件、className 大量覆盖和可上升抽象的业务组件。

Final Standard

优质组件系统应达到：设计变量统一、基础组件稳定、复合组件高频复用、业务组件边界清晰、页面只做编排、样式不随意散落、状态覆盖完整、文档和示例齐全、测试覆盖关键行为、长期演进有治理机制。
